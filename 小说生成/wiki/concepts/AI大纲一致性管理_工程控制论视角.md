# 工程控制论视角下的AI大纲一致性管理

> **核心课题**：如何让AI在生成大纲时前后内容不矛盾、时间线不冲突。
>
> **核心思路**：把小说大纲生成当作一个控制系统来设计——AI是被控对象，状态记录是反馈回路，人类主编是控制器。一致性不是"要求AI不犯错"，而是"建立一个系统，让AI的每一个输出都被检测、校正、归档"。
>
> **工程控制论的核心启示**：稳定性不是靠控制精度实现的，而是靠负反馈回路实现的。不要试图让AI一次做对，而要让AI每次犯错后都能被纠正。

---

## 一、为什么AI大纲生成天然会产生矛盾

### 1.1 AI缺乏"世界观锚定感"

人类写大纲时，脑中有一个隐含的"世界观数据库"——关于这个世界的一切假设都天然互相锁定。但AI每次生成新内容时，是从它对"这个世界"的整体理解中采样，而这个"整体理解"是模糊的概率分布，不是精确的规则库。

结果是：**同一个世界观设定，AI在不同章节生成时可能采纳了不同的子集**。

具体表现：

| 问题类型 | 典型案例 |
|---------|---------|
| 力量体系漂移 | 前文写"筑基期无法御剑飞行"，后文让筑基期角色御剑追击 |
| 人物状态失忆 | 前文写"林凡经脉尽断"，后文写"林凡如往常一样全力出拳" |
| 时间线错乱 | 写完"三年后"，下一章回到"第二天" |
| 地理信息矛盾 | 前文写"青云宗在北域"，后文写"从青云宗到南疆只需三天" |
| 关系前后不一 | 前文写"王二是林凡的死敌"，后文写"王二拍了拍林凡的肩膀" |

### 1.2 AI的"上下文遗忘"特性

AI的上下文窗口（Context Window）有限，超出范围的内容AI会"遗忘"——不是真的忘记，而是这部分的权重在生成新内容时变得极低。这就是为什么长篇大纲做到后期，前期的细节会越来越走样。

### 1.3 没有"写入确认"机制

人类写手写完一段，会自动检查这段是否与前文矛盾。AI不会——AI只关心"这一段好不好"，不关心"这一段和前面的关系对不对"。

## 二、控制论的基本框架与小说大纲的对应

### 2.1 经典控制框图

```
                    ┌──────────────┐
  设定值（大纲目标）→│   控制器     │→控制量（生成指令）
                    │  （主编）     │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │   被控对象   │
                    │  （AI模型）  │
                    └──────┬───────┘
                           │ 输出（大纲内容）
                    ┌──────▼───────┐
                    │   反馈环节   │
                    │ （一致性检查）│
                    └──────┬───────┘
                           │ 偏差信号
                      返回控制器
```

### 2.2 各环节在小说大纲中的对应

| 控制论术语 | 小说大纲对应 | 说明 |
|-----------|------------|------|
| **被控对象** | AI大纲生成模型 | 产生大纲内容的AI |
| **控制器** | 人类主编 | 做决策、给指令、批准内容 |
| **设定值** | 大纲目标文件 | 包含世界观/人物/力量体系/时间线 |
| **输出量** | 生成的大纲内容 | AI实际输出的章节/设定/情节 |
| **反馈环节** | 一致性检验机制 | 检查输出是否与设定值一致 |
| **偏差信号** | 一致性检验报告 | 列出发现的不一致之处 |
| **执行机构** | 修正指令 | 让AI修正不一致内容的指令 |

### 2.3 负反馈控制的核心公式

```
新的生成指令 = 原始生成指令 + K × （设定值 - 实际输出偏差）

其中 K 是反馈增益（修正力度）
```

**应用到小说大纲**：
```
下一段生成指令 = 原始情节需求 + （前文状态 - 当前输出偏差）
```

即：**每次生成新内容前，必须把"前文状态"和"新内容"之间的偏差作为约束条件加入Prompt**。

## 三、状态追踪系统：建立小说世界的"数字孪生"

### 3.1 什么是"数字孪生"

在工业控制中，数字孪生是指一个与真实物理系统完全同步的数字镜像。应用到小说大纲：**为你的小说世界建立一个精确的数据库，里面记录了这个世界的所有状态变量，AI生成新内容前必须查询这个数据库**。

### 3.2 状态变量的五大类别

```
小说状态变量体系
├── 角色状态
│   ├── 生理状态（伤势/疲劳/状态）
│   ├── 心理状态（当前情绪/心理变化轨迹）
│   ├── 能力状态（当前实力/金手指/技能）
│   └── 关系状态（与其他角色的当前关系）
├── 世界状态
│   ├── 势力格局（各势力的当前关系/实力对比）
│   ├── 资源分布（关键资源的所在）
│   └── 事件进展（主线/支线的当前节点）
├── 时间线状态
│   ├── 当前时间点（相对起点的时间）
│   ├── 时间跨度（某事件到某事件的时长）
│   └── 季节/时代背景
├── 地理状态
│   ├── 当前位置（各角色的当前位置）
│   ├── 地理距离（关键地点间的距离/耗时）
│   └── 势力范围边界
└── 设定状态
    ├── 力量体系规则（等级定义/限制条件）
    ├── 金手指规则（来源/限制/成长路径）
    └── 核心设定约束（哪些是绝对不能违反的）
```

## 四、一致性检验的三个层次

### 4.1 前馈检验（生成前）

**在调用AI生成新内容之前，主动把"前文状态"注入Prompt**。

这是最有效的检验层次——治未病，在错误发生之前就预防它。

#### 4.1.1 状态注入Prompt模板

```
## 当前状态快照（请在生成时严格遵守）

### 时间节点
当前是[故事开始后第X天]，[具体日期/节气]，[白天/夜晚]。

### 角色状态
- 林凡：筑基中期，右臂骨裂未愈（需注意：不能写他用右手发力）
- 王二：筑基后期，与林凡是死敌关系（已公开化）

### 力量限制
- 筑基期不能御剑飞行
- 金手指每日使用上限3次

### 前文已埋伏笔
- F001（Ch3神秘玉佩）：本章暂不涉及，如需涉及请严格遵循Ch3设定

### 本章绝对禁止出现的描写
- 林凡用右手全力出拳
- 筑基期角色御剑飞行
- 出现"金丹期"或以上的修炼者
```

#### 4.1.2 生成后检验（Post-Generation Check）

AI生成完内容后，用以下清单逐项检验：

**一致性检验三问**：

```
问1：时间线是否一致？
→ 上一章结束是什么时间？本章开始是什么时间？跨度合理吗？

问2：人物状态是否一致？
→ 本章人物的身体状态/心理状态/能力状态与上一章结尾是否衔接？

问3：世界观规则是否一致？
→ 本章出现的任何新设定/能力/物品是否与已确立的规则冲突？
```

#### 4.1.3 跨章一致性审计（定期）

每隔20-30章，进行一次"全局一致性扫描"：

```
扫描范围：最近生成的20章
扫描维度：
□ 角色状态链（从Ch1到当前，每章身体/心理/能力是否有断层）
□ 时间线审计（所有时间跳跃是否衔接，所有"三年后"类跳转是否有足够铺垫）
□ 地理一致性（所有地点描写是否与前文一致，距离是否自洽）
□ 势力关系演变（各角色之间的关系变化是否有逻辑链）
□ 伏笔状态（已埋伏笔是否在正确节点回收，有无遗漏）
□ 力量体系稳定性（升级/降级是否有充分铺垫）
```

### 4.2 时间线一致性的专项管理

时间线冲突是最常见也最难修复的问题。需要建立专项管理。

#### 4.2.1 时间线追踪的核心原则

| 原则 | 说明 |
|------|------|
| **相对时间 > 绝对时间** | 记录"从某事件到某事件用了多久"，而不是"发生在某年某月" |
| **有锚点事件** | 每个时间段有一个"锚点事件"作为参照（如"青云宗外门大比"） |
| **时间标记每次必填** | 每章生成后，必须记录本章结束时的时间点 |
| **跳跃必须有交代** | 任何时间跳跃（"三年后"）必须明确写出跳跃期间发生的事 |

#### 4.2.2 时间线审计脚本（概念）

```
# 时间线一致性检查伪代码

时间跳跃 = [章节中所有包含"年后"/"月后"/"天后"的句子]
for 每个时间跳跃 in 时间跳跃列表：
    检查跳跃幅度是否合理
    检查跳跃期间是否有交代
    检查跳跃后的状态是否与跳跃前逻辑衔接
```

## 五、闭环控制工作流：状态追踪驱动的AI大纲生成

### 5.1 标准工作流程（七步闭环）

```
步骤1：读取状态库
  ↓
步骤2：注入当前状态到Prompt
  ↓
步骤3：AI生成内容
  ↓
步骤4：一致性预检验（人工/AI）
  ↓
  ├─ 通过 → 步骤5
  └─ 不通过 → 步骤3'（强制AI基于错误反馈重写）
  ↓
步骤5：更新状态库
  ↓
步骤6：进入下一轮
  ↓
步骤7：定期跨章一致性审计
```

### 5.2 步骤详解

#### 步骤1：读取状态库

**每次调用AI生成新章节/新情节前，必须先执行这个动作**：

```
请读取以下状态文件：
[粘贴角色状态表]
[粘贴时间线状态]
[粘贴设定冲突登记表]

这些是当前的世界状态，请在新内容生成时严格遵守这些约束。
```

#### 步骤3→4：生成后一致性预检验Prompt

```
## 一致性检验任务

以下是新生成的内容，请检查是否存在以下问题：

### 时间线检查
- 本章开始时间是否与上一章结束时间衔接？
- 是否有时间跳跃？跳跃是否合理？
- 时间跨度与事件描述是否匹配？

### 角色状态检查
- [角色名]的身体状态是否与上一章结尾一致？
- [角色名]的心理状态是否有合理的变化原因？
- [角色名]的能力表现是否与当前实力等级匹配？

### 世界观规则检查
- 本章出现的设定是否与已确立的规则冲突？
- 力量体系表现是否自洽？
- 地理/距离信息是否前后一致？

请逐项列出发现的问题，并给出修改建议。
如无问题，请明确回复"通过检验"。
```

#### 步骤5：更新状态库

**这是整个控制回路中最重要的一步，也是最容易被忽略的一步**。

每次生成的内容经过检验确认后，必须同步更新状态库：

```markdown
## 状态库更新日志

### Ch19 结束时更新

**更新时间**：Ch19完成，检验通过后

**角色状态变更**：
- 林凡：右臂骨裂 → 右臂痊愈（Ch19获得奇遇）
- 林凡：实力等级 筑基中期 → 筑基中期（Ch19有小突破但未升级）

**时间线更新**：
- Day 22 → Day 23

**新埋伏笔**：
→ F004：Ch19出现的神秘女子（将在Ch35回收）

**新确立设定**：
✓ 青云宗内门弟子每月可领50灵石（Ch19明确）
```

### 5.3 分层控制：三种不同的控制粒度

根据大纲生成的不同阶段，采用不同精度的控制：

| 阶段 | 控制粒度 | 状态注入频率 | 检验频率 |
|------|---------|------------|---------|
| **粗纲阶段**（章节目录） | 每卷更新一次状态 | 每10章注入一次 | 每20章审计一次 |
| **细纲阶段**（章纲） | 每章更新一次状态 | 每章注入 | 每5章检验一次 |
| **正文阶段**（章节正文） | 每章内有多个状态点 | 每个场景前注入 | 每个场景后检验 |

## 六、AI大纲生成的反馈抑制器设计

### 6.1 反馈抑制器的概念

工程控制论中的"抑制器"（Stabilizer）是为了防止系统振荡/失控而设计的。应用到AI大纲生成：**当AI连续多次在同一类问题上出错时，抑制器会提高该类错误的检测强度**。

### 6.2 错误模式识别与强化检验

建立错误模式登记：

```markdown
## AI常见错误模式与强化检验

### 模式1：力量体系失忆
触发条件：连续2次在力量体系上出现矛盾
强化措施：在状态注入时增加力量体系限制的字号/强调
检查提示："筑基期绝对不能[具体行为]，请严格检查"

### 模式2：角色关系失忆
触发条件：任意1次出现关系矛盾
强化措施：每次生成前必须输出"角色关系状态表"
检查提示："[角色A]和[角色B]当前是[关系]，请确保不出现矛盾的互动"

### 模式3：时间线跳跃失控
触发条件：任意1次时间线矛盾
强化措施：每次生成前必须明确"当前时间点"
检查提示："请在生成后确认时间线是否衔接"
```

### 6.3 抑制器的自动升级机制

```
第1次某类错误 → 警告（提醒Prompt中加入一条注意）
第2次同类错误 → 强化（在状态注入中明确加入限制条款）
第3次同类错误 → 强制（给出必须遵守的具体规则，禁止AI自行决定）
第4次同类错误 → 中断（停止生成，要求人类主编介入处理）
```

## 七、设定冲突的应急处理机制

### 7.1 发现冲突后的处理流程

```
发现不一致
    ↓
确认影响范围（是孤立事件还是系统性矛盾）
    ↓
┌─────────────────┬─────────────────┐
│  孤立事件        │   系统性矛盾     │
│  （单章问题）     │  （跨章节问题）   │
├─────────────────┼─────────────────┤
│ 在该章节内修正    │ 需要重建状态库    │
│ 不影响前后章节    │ 从矛盾点开始回溯  │
│ 快速处理         │ 重新生成相关章节  │
└─────────────────┴─────────────────┘
    ↓
更新设定冲突登记表（记录本次冲突，避免重现）
    ↓
继续生成后续内容
```

### 7.2 设定冲突登记表示例

```markdown
## 设定冲突记录

### 冲突#001（已修复）
- 发现时间：Ch15
- 冲突描述：Ch15写"林凡经脉尽断"，但Ch14已写"林凡服用回春丹痊愈"
- 冲突原因：Ch14的生成没有查询状态库
- 处理方式：确认回春丹的设定在Ch14确立，以Ch14为准，Ch15重写
```

## 八、工程控制论核心原则在AI大纲生成中的应用

### 原则一：不要相信AI的"自觉"，要建立强制性的状态查询机制

**控制论启示**：开环系统（无反馈）是不稳定的。不要假设AI会主动保持一致性，要**强制**AI在每次生成前查询状态库。

**操作方法**：
- 把状态库查询变成Prompt的固定第一步
- 没有状态库查询的Prompt是不完整的Prompt
- 在工具/Skill层面强制要求状态库注入

### 原则二：状态估计的精度决定控制的上限

**控制论启示**：控制精度永远受限于状态估计精度。如果你自己都不清楚"当前世界的状态"，AI更不可能知道。

**操作方法**：
- 每次生成后立刻更新状态库，不能拖延
- 状态库的更新必须精确到具体数值，不能模糊
- 定期做全局状态审计，确保没有积累误差

### 原则三：控制回路必须闭合，不能有去无回

**控制论启示**：没有闭合的回路不是控制系统，是开环系统。AI生成内容后，必须有检验环节，检验结果必须反馈到下一次生成。

**操作方法**：
- 每次AI生成后，必须跟一个"一致性检验"步骤
- 检验结果必须以"修改指令"的形式反馈给AI
- 没有经过检验的内容不能进入状态库

### 原则四：积分项是系统不稳定的主要来源

**控制论启示**：积分控制项会让小误差累积成大偏差。应用到小说：**每次小的矛盾如果不处理，会在长篇中累积成根本无法修复的系统性矛盾**。

**操作方法**：
- 任何小的矛盾必须立即处理，不能"暂时忽略"
- 每周/每20章做一次历史矛盾扫描
- 发现积累性偏差必须回溯修正，不能"将错就错"

### 原则五：抗积分饱和——不要让AI在错误方向上走太远

**控制论启示**：积分饱和是指控制器在错误方向上累积了太多"积分"，导致即使纠正方向系统也要很久才能恢复。应用到小说：**AI在某个错误设定上写了很多章后，即使发现错误，修正成本也极高**。

**操作方法**：
- 控制生成节奏，不要让AI在没有被检验的情况下连续生成太多内容
- 每生成5-10章必须做一次一致性审计
- 发现系统性错误，宁可回退几章重写，也不要"将错就错"

## 十、总结：工程控制论对AI大纲生成的本质启示

### 十个核心要点

```
1. 一致性是系统问题，不是AI能力问题
   → 不要试图让AI"更聪明"，要建立一个"更严格的系统"

2. 状态库是一致性的根本前提
   → 没有精确的状态库，AI不可能保持一致性

3. 一致性检验必须强制闭合
   → 没有检验的生成是开环，开环必然漂移

4. 状态更新必须立即执行
   → 延迟更新状态库等于积累系统误差

5. 错误必须立即处理
   → 小矛盾的积累会摧毁整个系统

6. 前馈优于反馈
   → 在生成前注入约束 > 生成后检验修正

7. 控制粒度要与生成阶段匹配
   → 粗纲粗控，细纲精控，正文最精控

8. 抑制器防止同类错误重复发生
   → 同一类错误出现两次就必须升级控制强度

9. 定期全局审计防止积累误差
   → 每20-30章必须做一次跨章一致性扫描

10. 回退优于将错就错
    → 发现系统性错误，宁可回退重写
```

### 控制系统与大纲生成的完整对应

| 控制论概念 | 大纲生成应用 |
|-----------|------------|
| 被控对象 | AI大纲生成模型 |
| 控制器 | 人类主编（决策+批准） |
| 执行机构 | AI（执行生成指令） |
| 被控量 | 大纲内容质量与一致性 |
| 设定值 | 世界观设定/状态库 |
| 测量元件 | 一致性检验机制 |
| 反馈回路 | 检验→修正→更新状态库→下一轮生成 |
| 负反馈 | 每次生成都受到前文状态的约束 |
| 积分饱和 | 错误积累后难以纠正 |
| 抑制器 | 错误模式识别与强化检验 |
