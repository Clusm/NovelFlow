# 网文创作四人组·分级技能调用规则

> **严格遵循分级设定，禁止默认自由发挥，禁止OOC，禁止简化设定**

---

## 一、核心禁令（必须严格遵守）

### 🚨 绝对禁止条款

| 禁止项 | 说明 | 后果 |
|--------|------|------|
| **禁止默认自由发挥** | 任何创作必须先调用对应层级的子技能规则 | 视为违规 |
| **禁止OOC** | 禁止角色做出违背既定人设的行为/对话 | 视为OOC |
| **禁止简化设定** | 禁止省略/简化既定的世界观/人设/规则设定 | 视为违规 |
| **禁止跳过层级** | 必须从L1→L2→L3→L4逐级调用 | 视为跳级 |
| **禁止同级调用** | 禁止在未完成当前层级任务前调用同级其他技能 | 视为越权 |

### 🚨 创作禁令

| 禁止项 | 说明 |
|--------|------|
| **禁止脱离大纲** | 任何章节创作必须严格遵循章纲 |
| **禁止人设崩塌** | 角色行为必须符合既定的性格/能力/立场 |
| **禁止战力崩溃** | 战力提升必须有过程，战力差距必须有原因 |
| **禁止伏笔失踪** | 埋设的伏笔必须有回收计划，禁止烂尾 |
| **禁止信息突兀** | 角色知道的信息必须有来源和传递过程 |
| **禁止单文件超限** | 单个SKILL.md文件不得超过AI完整读取上限（约2000行） |
| **禁止技能树不完整** | 学习新技能必须同步更新技能树，禁止只更新SKILL.md不更新技能树 |

## 二、AI读取限制规则

> **单个SKILL.md文件必须控制在AI完整读取上限内，超限后自动拆分为L5子节点**

### 📏 文件大小限制

| 限制类型 | 上限 | 处理方式 |
|---------|------|---------|
| 单文件行数 | ≤2000行 | 超出自动拆分 |
| 单文件字符数 | ≤10万字符 | 超出自动拆分 |
| 单L4 rules数组 | ≤20条规则 | 超出自动拆分 |

### 🔄 自动拆分规则

```
文件超限检测
    ↓
判断超限原因：
├─ L4 rules数量过多 → 拆分为L5_XXX_a, L5_XXX_b
├─ L3内容过多 → 拆分为多个L3
└─ L2内容过多 → 拆分为多个L2
    ↓
执行拆分：
1. 创建新L5节点存储超限内容
2. 保持父节点引用完整性
3. 更新版本号
4. 同步更新技能树
```

### 📝 新技能学习流程

```
学习到新技能
    ↓
判断技能类型 → 确定归属L2分支
    ↓
检查目标L3节点是否超限
├─ 未超限 → 添加到现有L3
└─ 已超限 → 创建新的L3或L4节点
    ↓
更新技能树JSON（必须）
更新SKILL.md引用（必须）
更新版本号（必须）
```

## 三、分级调用规则

### 调用层级结构

```
L1_root（根节点）
    ↓ 必须先调用
L2_XX（一级分支）
    ↓ 必须先调用
L3_XXX（二级分支）
    ↓ 必须先调用
L4_XXXX（三级分支/具体规则）
```

### 调用流程

```
创作任务
    ↓
识别任务类型 → 确定属于哪个L2分支
    ↓
确定具体子技能 → 确定属于哪个L3分支
    ↓
调用具体规则 → 读取对应L4的rules数组
    ↓
严格遵循rules执行
    ↓
完成后返回上级，继续下一个任务
```

## 三、各角色调用规则

### 网文架构师调用规则

```
任务：架构设计
    ↓
[世界观类任务] → 调用 L2_world → 对应L3 → 对应L4
    ↓
[大纲架构类任务] → 调用 L2_outline → 对应L3 → 对应L4
    ↓
[角色架构类任务] → 调用 L2_characters → 对应L3 → 对应L4
    ↓
[力量体系类任务] → 调用 L2_system → 对应L3 → 对应L4
```

**示例：设计一个配角**
```
1. 调用 L2_characters（角色架构）
2. 调用 L3_supporting（配角设计）
3. 调用 L3_memory_point（记忆点设计）
4. 严格遵循 L4_memory_point 的rules：
   - 每个配角必须有1-2个标志性特征
   - 一句话能让人记住
   - 不能是工具人
```

### 网文创意总监调用规则

```
任务：创意策划
    ↓
[差异化定位任务] → 调用 L2_positioning → 对应L3 → 对应L4
    ↓
[钩子设计任务] → 调用 L2_hooks → 对应L3 → 对应L4
    ↓
[冲突设计任务] → 调用 L2_conflicts → 对应L3 → 对应L4
    ↓
[创新突破任务] → 调用 L2_innovation → 对应L3 → 对应L4
```

**示例：设计一个开篇钩子**
```
1. 调用 L2_hooks（钩子设计）
2. 调用 L3_golden_three（黄金三章设计）
3. 调用 L3_natural_hooks（自然钩子设计）
4. 严格遵循 L4_natural_hooks 的rules：
   - 信息不对等钩子/动作中断钩子/对话中断钩子/旧钩回咬
   - 不能用"一惊一乍"式的刻意悬念
```

### 网文原创作家调用规则

```
任务：原创写作
    ↓
[写作基础任务] → 调用 L2_writing → 对应L3 → 对应L4
    ↓
[人物写作任务] → 调用 L2_characters → 对应L3 → 对应L4
    ↓
[场景写作任务] → 调用 L2_scenes → 对应L3 → 对应L4
    ↓
[对话写作任务] → 调用 L2_dialogue → 对应L3 → 对应L4
```

**示例：写一段对话**
```
1. 调用 L2_dialogue（对话写作）
2. 调用 L3_subtext（潜台词设计）
3. 严格遵循 L4_subtext 的rules：
   - 角色嘴上说的不等于心里想的
   - 表面话必须有潜台词
   - 让读者有发现感
```

### 网文审核官调用规则

```
任务：审核质检
    ↓
[合规审核任务] → 调用 L2_compliance → 对应L3 → 对应L4
    ↓
[质量审核任务] → 调用 L2_quality → 对应L3 → 对应L4
    ↓
[逻辑审核任务] → 调用 L2_logic → 对应L3 → 对应L4
    ↓
[AI检测任务] → 调用 L2_ai_detect → 对应L3 → 对应L4
```

**示例：审核一段文字的AI味**
```
1. 调用 L2_ai_detect（AI检测）
2. 调用 L3_word_freq（词频异常检测）
3. 严格遵循 L4_ai_adverbs 的rules：
   - 检测'非常''极其''不可思议'等词
   - 超过4次必须修改
   - 用具体描写替代
```

### 拆书分析师调用规则

```
任务：拆书分析
    ↓
[选书任务] → 调用 L2_preparation → L3_selection → L4_criteria
[阅读任务] → 调用 L2_preparation → L3_reading → L4_methods
[拆解任务] → 调用 L2_analysis → 对应维度L3 → L4_dimension
[提炼任务] → 调用 L2_extraction → L3_formulas → L4_template
[输出任务] → 调用 L2_output → L3_report → L4_format
```

**强制执行**：
1. 严格遵循 L2_preparation → 选书标准（榜单/匹配/质量/时效）
2. 严格遵循 L2_analysis → 四维度拆解（人设/节奏/结构/卖点）
3. 严格遵循 L2_extraction → 套路提炼公式
4. 拆解后必须提供可应用的套路模板
5. 必须完成完整的拆书报告（六板块）

## 四、OOC判定标准

### 什么是OOC？

**OOC = Out Of Character（角色脱离）**

指角色做出了违背既定人设、性格、能力、立场的行为或对话。

### OOC判定清单

```
□ 角色说话方式是否与既定性格一致？
□ 角色决策是否符合其能力范围？
□ 角色情绪反应是否符合情境？
□ 角色立场是否前后矛盾？
□ 角色行为是否符合社会身份？
```

### 常见OOC示例

| 场景 | OOC行为 | 正确行为 |
|------|---------|---------|
| 高冷角色 | 突然说大量话痨台词 | 保持惜字如金 |
| 懦弱角色 | 突然正面硬刚 | 保持退缩但有底线 |
| 穷苦角色 | 用富人思维行事 | 保持穷人思维逻辑 |
| 聪明角色 | 犯低级错误 | 保持聪明人设 |

## 五、禁止简化设定判定

### 什么是"简化设定"？

在创作过程中省略、简化、或遗忘既定的世界观/人设/规则设定。

### 简化设定判定清单

```
□ 世界观设定是否完整呈现？
□ 等级体系是否严格遵循？
□ 势力关系是否按设定执行？
□ 角色能力是否在范围内？
□ 时间线是否连贯一致？
□ 伏笔是否按计划埋设/回收？
```

### 常见简化设定示例

| 类型 | 简化行为 | 正确行为 |
|------|---------|---------|
| 等级简化 | 越级打败比自己高3级的对手，无任何解释 | 必须有特殊原因（秘术/机缘/策略） |
| 势力简化 | 忽视势力关系，直接和敌对势力合作 | 必须有铺垫和转折 |
| 能力简化 | 无条件使用金手指 | 必须有使用条件和代价 |

## 六、违规处理

### 违规等级

| 等级 | 违规类型 | 处理方式 |
|------|---------|---------|
| **P0** | 核心禁令违反（禁止OOC/简化设定） | 必须立即修正 |
| **P1** | 分级调用违反（跳级/同级调用） | 必须回退重新调用 |
| **P2** | 规则遵循不严（部分rules未执行） | 必须补充完善 |

### 修正流程

```
发现违规
    ↓
识别违规类型
    ↓
P0 → 立即回退，修正OOC/简化问题
P1 → 回退到对应层级，重新调用
P2 → 补充缺失的rules执行
    ↓
完成修正
    ↓
重新审核
```

## 九、同步更新机制（强制执行）

> **LLM Wiki技能树 与 四人组SKILL.md 必须同步更新，不允许出现内容不一致**

### ⚡ 同步更新原则

| 原则 | 说明 |
|------|------|
| **即时同步** | 任何一方更新，另一方必须立即同步 |
| **内容一致** | L4 rules内容必须完全一致 |
| **版本统一** | 版本号同步更新 |
| **时间戳同步** | updated_at必须一致 |

### 🔄 同步更新触发条件

```
触发条件：
1. LLM Wiki技能树任意L4规则更新
2. 四人组SKILL.md任意技能内容更新
3. 新增/删除任意子技能
4. 规则描述变更
```

### 📝 同步更新流程

```
发现需要更新
    ↓
判断更新来源：
├─ LLM Wiki技能树更新 → 同步到四人组SKILL.md
└─ 四人组SKILL.md更新 → 同步到LLM Wiki技能树
    ↓
执行双向同步：
1. 更新目标文件的对应L4 rules
2. 更新版本号（+0.0.1）
3. 更新时间戳
4. 记录同步日志
    ↓
验证一致性：
- L4 rules内容完全一致
- 版本号一致
- 时间戳一致
```

### 📋 同步检查清单

每次调用技能树前必须检查：
```
□ LLM Wiki版本号 = 四人组SKILL.md版本号？
□ L4 rules内容是否一致？
□ 更新时间戳是否一致？
□ 如发现不一致，立即执行同步更新
```

### ⚠️ 同步违规处理

| 违规类型 | 处理方式 |
|---------|---------|
| **未同步更新** | 视为严重违规，必须立即同步 |
| **版本不一致** | 以LLM Wiki技能树为准，立即同步SKILL.md |
| **内容不一致** | 立即对比并修正不一致内容 |

### 🔢 版本号管理

| 文件 | 版本号格式 | 同步规则 |
|------|----------|---------|
| LLM Wiki | x.y.z | 与各技能树同步 |
| tree_architect.json | x.y.z | 与index.json同步 |
| tree_director.json | x.y.z | 与index.json同步 |
| tree_writer.json | x.y.z | 与index.json同步 |
| tree_auditor.json | x.y.z | 与index.json同步 |

**版本号递增规则**：
- x：大版本更新（架构重构）
- y：功能更新（新增L2/L3）
- z：小修小改（L4 rules调整）
