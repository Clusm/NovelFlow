# 网文创作核心工作流程规则

## 🚨 强制执行规则

### ⚠️ 铁则：未续写 = 不满意，禁止复用

**【全局强制指令】**
- **定义**："没有续写下去" = 用户不满意，创作失败
- **禁止**：失败的项目/框架/文笔不得复用
- **要求**：每次新创作必须重新思考，不能重复已有模式
- **例外**：只有"用户明确要求续写"才可继续，否则视为不满意

**【新项目设定隔离规则】**
| 项目 | 规则 |
|------|------|
| **适用场景** | 新小说创作项目启动时 |
| **核心原则** | 禁止复用之前任何项目的设定 |
| **允许范围** | 只有用户**明确要求**复用已有设定，才可使用 |
| **禁止项** | ❌ 禁止复用失败项目的世界观/力量体系/角色名/金手指/剧情框架 |
| **违规后果** | 视为创作失败，需彻底重新规划 |

**【解读】**
| 状态 | 含义 | 处理方式 |
|------|------|---------|
| 未开始新项目 | 用户在思考 | 等待用户指令 |
| 创作后未续写 | 用户不满意 | 禁止复用该框架/文笔 |
| 用户主动停止 | 失败 | 禁止复用，重新分析根源 |
| 多次同一模式失败 | 系统性问题 | 彻底反思而非修修补补 |
| 用户要求复用 | 明确授权 | 可复用，但需重新验证适配性 |

**触发条件：只要创作小说或跟小说相关的一切任务，必须按顺序调用以下技能！**

### ⚠️ 铁则：作家✘审核协同联动（全局通用）

**【全局强制指令·适用所有项目】**
- **定义**：作家写完任何内容 = 自动触发审核 = 全程无需用户确认
- **范围**：本规则适用于**所有小说创作项目**，无例外
- **禁止**：禁止跳过审核流程、禁止未经审核发布、禁止手动关闭联动

**【强制要求】**
| 触发场景 | 执行动作 | 强制级别 |
|---------|---------|---------|
| 作家写完单段 | 自动触发审核官 | ⚠️ 强制 |
| 作家写完单章 | 自动触发审核官 | ⚠️ 强制 |
| 作家写完多章 | 自动触发审核官 | ⚠️ 强制 |
| 审核发现问题 | 自动精修+重审 | ⚠️ 强制 |
| 精修完成 | 自动重新审核 | ⚠️ 强制 |
| 审核通过 | 自动通知发布 | ✅ 执行 |

**【协同流程】**

```
作家创作完成 → 【自动触发·无例外】 → 审核官接收
    ↓
审核官 → P0/P1/P2/P3质检
    ↓ 【发现问题】
自动精修优化 → 重新审核
    ↓ 【循环】
章节达标 → 发布（自动通知用户）
```

**【问题优先级】**

| 等级 | 类型 | 处理方式 | 循环次数 |
|------|------|---------|---------|
| P0 | 致命（合规/AI痕迹） | 立即修改 | 无限循环直到P0通过 |
| P1 | 重要（逻辑/人设/战力） | 精修后重审 | 直到P1+P0全部通过 |
| P2 | 优化（节奏/爽点） | 建议修改 | 直到通过或用户override |
| P3 | 建议（伏笔/细节） | 可选优化 | 可跳过 |

**【核心原则】**
- ✅ 全程无需用户确认
- ✅ 不动作家核心剧情与原生文风
- ✅ 只做精修打磨，不做大幅改写
- ❌ 禁止跳过审核
- ❌ 禁止手动关闭协同联动

### ⚠️ 铁则：作家完成后必须自动触发审核

**【强制要求】**
- 原创作家完成任何形式的创作（含续写、单段、多段、单章、多章）后，**无需用户确认**，必须立即自动调用网文审核官进行质检
- 审核内容包括：合规安全、AI痕迹检测、剧情逻辑、人设一致性、战力稳定性、读者留存吸引力
- 即使只有一段话、一个问题、一个片段，也必须执行审核流程
- 审核未通过的章节不得发布，必须修改至通过后才能继续

**【发现问题自动优化机制】**
- 审核完成后，**无需用户确认**，网文审核官必须根据审核结果自动进行精修优化
- 优化优先级：P0（致命）→ P1（重要）→ P2（优化）→ P3（建议）
- 优化完成后**自动重新审核**，直至通过或达到可发布标准
- 优化过程中不动作者核心剧情与原生文风，只做精修打磨

**【自动触发机制】**
- **全程无需用户确认**：创作→审核→优化→再审核，整个流程自动执行，不询问用户
- 创作/续写完成后立即进入审核流程
- 审核完成后立即进入优化流程
- 优化完成后立即重新审核
- 循环直至章节达到可发布标准

```
原创作家完成创作
    ↓ 自动触发（强制执行，无需询问）
网文审核官 → 质检审核
    ↓ 发现问题
自动精修优化 → 重新审核
    ↓ 循环直至通过
发布（达标后自动通知用户）
```

### ⚠️ 铁则：Skill调用执行规范

**【强制执行流程】**

| 阶段 | 动作 | 说明 |
|------|------|------|
| **1. 调用前** | 确认禁止项 | 检查是否有违规项 |
| **2. 调用时** | 层级调用 | 必须L1→L2→L3→L4逐级调用 |
| **3. 执行中** | 对照检查 | 每一步对照skill检查清单 |
| **4. 执行后** | 自检验证 | 完成后对照禁止项和检查清单自检 |

**【层级调用规则】**

```
收到任务
    ↓ 【禁止项检查】
    ↓ 【违规？→ 停止，重新规划】
    ↓ 【无违规】
    ↓ 调用L1主skill
    ↓ 调用对应L2子skill
    ↓ 调用对应L3子skill（如有）
    ↓ 调用对应L4子skill（如有）
    ↓ 【执行中对照检查清单】
    ↓ 【执行后自检验证】
    ↓ 完成
```

**【执行后自检清单】**

```
□ 是否遵守了所有禁止项？
□ 是否完成了所有命令清单？
□ 是否执行了所有检查清单？
□ 输出是否符合skill要求？
□ 是否有遗漏的步骤？
□ 是否触发了下一个流程？
```

**【违规处理】**

| 违规类型 | 处理方式 |
|---------|---------|
| 跳过层级 | 必须返回上一层重新执行 |
| 违反禁止项 | 立即停止，修正后才能继续 |
| 跳过自检 | 必须补充自检才能输出 |
| 未完成清单 | 必须完成才能进入下一步 |

### ⚠️ 铁则：Skill自主拆分机制

**【AI友好语言描述标准】**

| 类型 | 要求 | 示例 |
|------|------|------|
| **结构化** | 使用表格/列表/流程图 | `Step 1: xxx → Step 2: xxx` |
| **简洁** | 每段≤3行 | 标题+内容 |
| **明确** | 动词开头 | `执行/检查/输出/调用` |
| **统一** | 格式一致 | `[触发条件]` `[执行动作]` |
| **分层** | 标题层级清晰 | ## → ### → #### |

**【模板结构】**

```markdown
# Skill名称

## 元数据
> 版本 | 来源 | 定位

## 执行流程
```
输入 → [Step 1] → [Step 2] → [Step 3] → 输出
```

## 命令清单

### 命令1
- 触发：xxx
- 输入：xxx
- 输出：xxx
- 禁止：xxx

### 命令2
...

## 检查清单
- [ ] 检查项1
- [ ] 检查项2
```

**【核心问题】**

| 问题 | 说明 |
|------|------|
| skill文件超限 | 超过200行限制，AI无法完整读取 |
| 不知道何时拆分 | 缺乏明确的拆分触发条件 |
| 拆分不规范 | 拆分的L2文件没有标准结构 |

**【AI行数限制】**

| 层级 | 行数限制 | 说明 |
|------|---------|------|
| 主skill (L1) | ≤200行 | AI完整读取的上限 |
| L2子skill | ≤200行 | AI完整读取的上限 |
| L3子skill | ≤200行 | AI完整读取的上限 |
| L4子skill | ≤200行 | AI完整读取的上限 |

**【自主拆分触发条件】**

| 触发条件 | 当前值 | 自动动作 |
|---------|-------|---------|
| 文件行数 | >200行 | 自动拆分 |
| 单个技能内容 | >100行 | 拆分子skill |
| 重复内容 | >50字 | 合并到共享文件 |

**【自主拆分决策树】**

```
检查文件行数
    ↓
【>200行？】
    ↓ 是
判断内容类型：
    ↓
【有独立技能模块？】
    ↓ 是 → 拆分为L2子skill
    ↓ 否
【内容是否重复？】
    ↓ 是 → 合并到共享方法论库
    ↓ 否
【是否有层级关系？】
    ↓ 是 → 拆分到对应层级
    ↓ 否
精简内容（删除冗余描述）
```

**【自主拆分标准流程】**

```
Step 1：检测文件行数
    ↓
Step 2：分析内容结构
    ↓
Step 3：判断拆分类型
    ↓
Step 4：创建L2/L3/L4文件夹
    ↓
Step 5：拆分内容到子skill
    ↓
Step 6：更新主skill引用
    ↓
Step 7：验证拆分后行数≤200
    ↓
Step 8：输出拆分报告
```

**【拆分类型判断】**

| 条件 | 拆分类型 | 目标位置 |
|------|---------|---------|
| 有多个独立技能 | 按技能拆分 | L2/{技能名}/SKILL.md |
| 有层级关系 | 按层级拆分 | L2/L3/L4 |
| 内容重复 | 合并到共享 | 共享方法论库/ |
| 整体臃肿 | 精简+保留核心 | 主文件 |

**【拆分报告格式】**

```markdown
## Skill自主拆分报告

### 日期：[YYYY-MM-DD]

### 检测结果
| 文件 | 行数 | 状态 |
|------|------|------|
| XXX/SKILL.md | 250行 | 超出50行 |

### 执行操作
| 操作 | 目标 | 状态 |
|------|------|------|
| 创建 | XXX/L2/技能A/SKILL.md | ✅ |
| 拆分 | 技能A内容 → L2/技能A | ✅ |
| 更新 | 主文件引用 | ✅ |

### 拆分后验证
| 文件 | 行数 | 状态 |
|------|------|------|
| XXX/SKILL.md | 180行 | ✅ |
| XXX/L2/技能A/SKILL.md | 85行 | ✅ |
```

**【禁止项】**

| 禁止 | 正确 |
|------|------|
| 文件超过200行不拆分 | 超限即拆分，无需询问 |
| 拆分后仍有文件超限 | 继续拆分直到全部≤200行 |
| 不更新引用关系 | 主文件必须引用L2/L3/L4 |
| 拆分不输出报告 | 每次拆分必须输出报告 |

**【自主拆分检查点】**

```
□ 每次调用skill前检查文件行数？
□ 超过200行立即拆分？
□ 拆分后验证所有文件≤200行？
□ 更新主skill引用关系？
□ 输出拆分报告？
□ 记录到拆分日志？
```

### ⚠️ 铁则：技能重复检测与去重机制

**【核心问题】**

| 问题 | 说明 |
|------|------|
| 技能重复 | 多个skill可能包含相同技能（如钩子设计、节奏把控） |
| 重复执行 | 重复技能会被重复调用，浪费时间 |
| 冲突风险 | 重复技能可能产生冲突的指令 |
| 无追踪 | 无法追踪已执行的技能 |

**【技能重复检测表】**

**覆盖范围**：主skill + L2/L3/L4子skill

**一、主skill之间重复（⚠️ 高风险）**
| 技能类型 | 所属skill | 去重规则 |
|---------|-----------|---------|
| 钩子设计 | 拆书分析师+创意总监 | 策划阶段创意总监主执行 |
| 节奏把控 | 拆书分析师+原创作家+审核官 | 原创作家主执行 |
| 人物塑造 | 架构师+拆书分析师+原创作家 | 架构师主执行 |

**二、L2子skill之间重复（⚠️ 中风险）**
| 技能类型 | 所属L2子skill | 去重规则 |
|---------|---------------|---------|
| 节奏平衡 | 原创作家节奏功+创意总监紧张功 | 原创作家节奏功主执行 |
| 爽点设计 | 原创作家节奏功+创意总监紧张功 | 创意总监主执行 |
| 人设设计 | 架构师人设功+原创作家人设功 | 架构师人设功主执行 |

**三、L3/L4子skill之间重复（⚡ 低风险）**
| 技能类型 | 所属L3/L4子skill | 去重规则 |
|---------|------------------|---------|
| 钩子设计 | 拆书反套路+创意自然钩子 | 创意自然钩子主执行 |
| AI检测 | 审核逻辑审核+逻辑审核L4 | L4主执行 |

**四、完整重复检测矩阵**
| | 架构师 | 拆书 | 创意 | 作家 | 审核 |
|---|--------|------|------|------|------|
| 架构师 | - | 人设(中) | 冲突(高) | 人设(中) | 世界观(中) |
| 拆书 | 人设(中) | - | 钩子(高) | 节奏(高) | 节奏(中) |
| 创意 | 冲突(高) | 钩子(高) | - | 爽点(中) | 爽点(中) |
| 作家 | 人设(中) | 节奏(高) | 爽点(中) | - | 节奏(中) |
| 审核 | 世界观(中) | 节奏(中) | 爽点(中) | 节奏(中) | - |

**【执行流程】**

```
调用skill前
    ↓
检查技能重复表
    ↓ 【有重复？】
    ↓ 是 → 执行去重机制
    ↓ 否 → 正常执行
    ↓
去重机制：
1. 识别重复技能
2. 确定主执行skill（核心职责优先）
3. 其他skill引用结果，不重复执行
4. 记录调用日志
5. 触发自动清理/优化（如需要）
```

**【自动清理机制】**

| 重复类型 | 处理方式 | 触发条件 |
|---------|---------|---------|
| L3/L4子skill内容重复 | 合并到共享文件 | 重复内容超过50字 |
| L2子skill功能重复 | 标记主/从关系 | 同类技能出现2次以上 |
| 主skill职责重复 | 更新到去重表 | 冲突次数超过3次 |

**【自动优化机制】**

```
Step 1：识别重复内容
Step 2：评估重复程度（高/中/低）
Step 3：高重复 → 自动合并到共享文件
Step 4：中重复 → 标记主skill，引用关系
Step 5：低重复 → 记录但保留
Step 6：更新技能重复检测表
Step 7：输出优化报告
```

**【优化报告格式】**

```markdown
## 技能优化报告

### 日期：[YYYY-MM-DD]

### 发现的重复
| 重复技能 | 所属skill | 重复程度 | 处理方式 |
|---------|-----------|---------|---------|
| XXX | A+B | 高 | 已合并到共享文件 |
| YYY | C+D | 中 | 标记主skill |

### 执行的操作
- [ ] 合并：XXX → 共享方法论库/XXX.md
- [ ] 更新：技能重复检测表
- [ ] 引用关系：C → 主执行，D → 引用C

### 待处理
- [ ] XXX（等待确认）
```

**【禁止项】**

| 禁止 | 正确 |
|------|------|
| 静默跳过重复技能 | 必须记录并引用 |
| 重复执行相同技能 | 去重后只执行一次 |
| 冲突时不解决 | 必须确定优先级 |
| 不记录调用日志 | 每次调用必须记录 |
| 发现高重复不优化 | 必须触发自动清理机制 |

**【去重规则】**

| 场景 | 处理方式 |
|------|---------|
| 多个skill有相同技能 | 核心职责skill执行，其他引用结果 |
| 审核与创作冲突 | 以创作为准，审核提供建议 |
| 架构与创意冲突 | 架构优先，创意适配架构 |
| 检测到跳过重复 | 记录并预警，禁止静默跳过 |

**【调用日志格式】**

```markdown
## 技能调用日志

### 项目：[项目名]
### 日期：[YYYY-MM-DD]

| 调用时间 | Skill | 执行的技能 | 去重说明 |
|---------|------|------------|---------|
| HH:MM | 创意总监 | 钩子设计 | 拆书分析师已执行，引用其结果 |
| HH:MM | 拆书分析师 | 钩子设计 | 主执行 |
```

**【禁止项】**

| 禁止 | 正确 |
|------|------|
| 静默跳过重复技能 | 必须记录并引用 |
| 重复执行相同技能 | 去重后只执行一次 |
| 冲突时不解决 | 必须确定优先级 |
| 不记录调用日志 | 每次调用必须记录 |

### ⚠️ 铁则：联网学习执行规范

**【强制执行流程】**

| 阶段 | 动作 | 说明 |
|------|------|------|
| **1. 搜索前** | 明确学习目标 | 确定要找什么信息 |
| **2. 搜索中** | 分层搜索 | 先概览后深入，每个维度1-2次搜索 |
| **3. 筛选中** | 选择高价值页面 | 来源权威+内容相关+不重复 |
| **4. 深度阅读** | 提取关键信息 | 不要全文翻译，要提炼要点 |
| **5. 整合分析** | 形成结论 | 对照学习目标，检查是否找到答案 |
| **6. 输出报告** | 结构化呈现 | 发现/结论/建议/行动项 |

**【搜索原则】**

```
□ 搜索词精准，不模糊
□ 优先搜索一手来源（官方/专业媒体）
□ 避免重复来源（相似标题只选一个）
□ 每轮搜索后先展示中间结果，再决定是否继续
□ 有足够信息立即停止，不追求穷尽
```

**【深度阅读原则】**

```
□ 只读与学习目标相关的部分
□ 每页提取3-5个关键信息点
□ 标注来源URL以便追溯
□ 不追求读完全文，够用即可
```

**【整合分析原则】**

```
□ 对比不同来源的信息，找出共识和分歧
□ 分析信息的可信度和时效性
□ 提炼出可执行的建议
□ 识别信息缺口，明确后续行动
```

**【输出报告标准】**

```
□ 明确的学习目标
□ 搜索的中间发现
□ 关键来源列表（含URL）
□ 提炼的核心结论
□ 可执行的建议
□ 尚未解决的问题（如果有）
```

## 🏷️ 项目状态追踪系统（全局通用）

### 系统文件位置

```
projects/
├── _global/
│   ├── status_template.json    # 全局状态模板
│   └── status_rules.md         # 状态管理规则
└── [项目名]/
    └── status.json             # 项目状态文件
```

### 项目状态定义

| 状态值 | 含义 | 显示优先级 |
|--------|------|-----------|
| `planning` | 规划中 | 低 |
| `active` | 进行中 | 高 |
| `paused` | 暂停中 | 中 |
| `completed` | 已完结 | 低 |
| `abandoned` | 已放弃/烂尾 | 低 |

### 完结判断标准

标记为 `completed` 必须满足以下条件之一：
- ✅ 用户明确宣布完结（`userDeclared: true`）
- ✅ 所有故事线/大篇章已完成（`allArcsFinished: true`）
- ✅ 所有伏笔已回收（`allForeshadowingResolved: true`）
- ✅ 有明确的结局章节（`hasEndingChapter: true`）

### 放弃判断标准

标记为 `abandoned` 的条件：
- 用户主动放弃（"不写了"、"放弃"）
- 暂停超过60天未更新
- 连续3次创作失败

### 自动清理规则

| 触发条件 | 自动动作 |
|---------|---------|
| paused 超过60天 | → abandoned |
| 用户宣布放弃 | → abandoned |
| 完结标准满足 | → 询问是否 completed |
| 新项目创建 | → planning |

### 强制规则

1. **禁止隐藏状态**：所有项目必须有准确的状态值
2. **禁止僵尸项目**：paused 超过60天自动转为 abandoned
3. **禁止复用失败**：abandoned 项目的框架/文笔禁止复用
4. **INDEX.md 同步**：状态变更必须同步更新 INDEX.md
