# 网文大纲质量审查方法论

## 何时使用
当需要系统性审查小说大纲（5卷+/50章+），验证质量维度：爽点密度、人设轨迹、黄金开局、时间线一致性、伏笔回收。

## 核心约束
- 用 `execute_code` + `regex` 做机器检查，不用子代理
- 检查结果直接输出给用户，不写入文件
- 发现问题后 patch 精确修复，不重写

## 审查维度与脚本方法

### 1. 爽点密度（每10章统计一次）

```python
import re

# 检查4类爽点关键词在每章的出现
slap = bool(re.search(r'打脸|抽脸|怒斥|反击', chapter))
reveal = bool(re.search(r'震惊|真相|原来|恍然大悟', chapter))
sweet = bool(re.search(r'吻|牵手|拥抱|脸红|表白|求婚', chapter))
cold = bool(re.search(r'冷笑|眼底闪过寒光', chapter))
```

**番茄文爽点类型分类**：
- 打脸：反驳原话+表情变化+围观者反应（三要素缺一不可）
- 揭露：真相揭晓+当事人震惊
- 甜：肢体接触/情感表白/求婚
- 冷：主角冷对反派，气场碾压

**判断标准**：
- 黄金开局（Ch1-Ch10）：每10章打脸≥2，甜≥1
- 中段（Ch11-Ch200）：每10章爽点总数≥5
- 终局（Ch201-Ch250）：每10章打脸+揭露≥10（密集轰炸）

**异常信号**：
- 某10章段打脸=0且甜=0 → 节奏断层
- 某10章段打脸=0但甜=10 → 感情断档（需要打脸调剂）

### 4. 时间线一致性

```python
# 检查年份关键词的章节分布
year_positions = [(m.start(), m.group()) for m in re.finditer(r'20\d\d年', content)]
chapter_markers = [(m.start(), m.group(1)) for m in re.finditer(r'### Ch(\d+)', content)]

# 标注每个年份属于哪一章
for pos, year in year_positions:
    for i in range(len(chapter_markers)-1):
        if chapter_markers[i][0] <= pos < chapter_markers[i+1][0]:
            print(f"{year} @ Ch{chapter_markers[i][1]}")
            break
```

**时间线规则**：
- 年份只能递增（2014→2015→2016→2017），不允许倒退
- 重大事件（上线/融资/求婚）必须有年份标注
- 跨卷年份要一致（第三卷写了2014年求婚，第四卷就不能再写2014年）

## 实战发现（2026-04-25 她还在读大三项目）

**发现1：子智能体扩展会导致占位符残留**
- 子智能体扩展后的文件仍有"（约XXX字）"占位符
- 这些占位符是细纲的正常状态，不代表内容缺失
- 不要因为"有占位符"就误判为需要重写

**发现2：打脸密度和甜密度需要平衡**
- 第二卷Ch51-Ch100：甜=10但打脸=0 → 节奏失衡，需要在感情戏中插入打脸调剂
- 第三卷Ch101-Ch150：打脸=3但甜=1 → 太压抑，需要在复仇线中插入甜蜜喘息

**发现3：反派退场必须有交代**
- 林雅Ch1后消失无后续交代 → 需要在第二卷补一句结局
- 赵泰Ch50击溃后消失 → 合理（已完成使命）

**发现4：室友线必须是情感锚点**
- 室友必须在每个重大事件节点出现（接管公司/百富榜/求婚）
- 只在某一卷集中出现=室友线断裂

**发现5：patch残留bug——多字段内容的old_string必须包含完整字段**
- 当要替换的内容包含多个 `- **字段名**：xxx` 行时，只截取部分内容作为 old_string 会留下残留
- 示例：替换"剧情"字段时，只写部分剧情文字 → 残留"理。陈权走出来，语气平淡..."
- 修复方法：old_string 必须包含从 `- **剧情**：` 到下一个 `- **爽点` 之间的全部内容，不留边界
- 验证：每次 patch 后立即检查文件该区域，确认无残留字符

**发现6：新章节插入后必须同步更新相邻章节的章末钩子**
- 在 Ch132 后插入 Ch132b 时，Ch132 的原钩子"室友群聊..."已不再指向下一个物理章节
- 正确做法：先改 Ch132 的钩子指向 Ch132b，再插入新章节，Ch132b 的钩子指向原 Ch133
- 如果不提前改钩子就插入 → 钩子指向错乱，读者体验断裂

**发现7：占位符的精确含义（实测数据）**
- 实测数据：Volume 1 = 50处占位符（每章1处），Volume 2-5 = 0处
- 含义：Volume 1 的50章全部是"（约XXX字）"→ 这是细纲骨架，写正文时以场景/爽点/台词/钩子为提纲自行填充
- 后续补充的新章节（Ch70b/Ch132b/Ch132c/Ch134b）全部是完整剧情，不是占位符
- 结论：`(约XXX字)` 不代表内容缺失，而是细纲的正常表达方式

**发现8：爽点密度的"甜蜜区"识别**
- 实测数据：第二卷Ch51-Ch100，甜度10+的10章段有3个（Ch71-80/Ch91-100），打脸=0
- 含义：感情验证期全是甜戏是正常的，但如果全程无任何对立冲突（室友质疑=对立），会导致阅读疲劳
- 优化公式：在感情甜蜜期中，每20章应插入1个对立冲突事件（如室友质疑），避免"甜蜜沼泽"
- 本次修复：Ch70b（苏念反击小A）在甜蜜期中插入对立冲突，提升节奏感

**发现9：多卷大纲的"章节标题前缀不一致"问题（文道至尊项目）**
- 问题：同一项目中，V1-V4使用`### 第XXX章：`格式（Markdown三级标题），V5使用`第XXX章：`格式（无`###`前缀）
- 根本原因：subagent或不同扩展流程产生的文件格式不统一
- 审查方法：先用 `re.findall(r'### 第(\d+)章', content)` 和 `re.findall(r'第(\d+)章', content)` 分别扫描，确认各卷格式
- 修复方法：统一使用一种格式（推荐`### 第XXX章：`），在验证脚本中对不同格式分别用不同regex
- 验证脚本必须在同一脚本中对不同卷使用不同的章节匹配pattern

**发现10：大纲与实际细纲的"Act结构漂移"问题**
- 问题：D_分卷大纲中声明的Act范围（如"第三幕 Ch241-Ch300"）与实际细纲章节分布不符（实际Ch261-291是正文，Ch292-300是番外）
- 根本原因：写作过程中章节内容调整，但设计文档（D大纲）未同步更新
- 发现方法：提取所有`## 第X幕...（ChXXX-ChXXX）`与实际章节扫描结果对比
- 修复策略分两种：
  - 方案A（以D大纲为准）：调整细纲章节归属
  - 方案B（以实际细纲为准）：重写D大纲
  - 推荐方案B，因为实际写作内容更权威
- 实操：先用execute_code提取所有Act头范围并与实际章节分布对比，定位漂移点，再决定修复策略

**发现11：卷终章节重复标注问题**
- 问题：同一卷内出现两个章节都标"XX卷终"（如V1 Ch80和Ch100都标"第一卷终"，V3 Ch291和Ch300都标"第三卷终"）
- 根本原因：写作时分两次添加卷终内容，但忘记修改前一个卷终的标签
- 审查方法：扫描所有含"卷终"/"全书终"关键词的章节标题
- 修复原则：
  - 正文最后一章：标签=卷终/全书终
  - 中点章节：标签=中点/转折，不应用"卷终"
  - 番外章节：单独幕，不与正文卷终混淆
- 实操案例：
  - V1 Ch80（误标卷终）→ 改为"第一卷中点·寒门崛起"
  - V2 Ch145（误标卷终）→ 改为"第二卷中点·黎明将至"
  - V3 Ch291（真卷终）和Ch300（番外尾）→ Ch291保留"第三卷终·知识永恒"，Ch300改为"第三卷终·知识的黎明"以示区分

**发现12：patch失败时的文件级修复方法**
- 问题：patch报错"old_string not found"，但search_files能找到内容
- 原因：文件中存在不可见字符（换行符差异、多余空格、全角/半角冒号）导致精确匹配失败
- 解决策略（按优先级）：
  1. 扩大old_string范围，确保唯一性
  2. 改用 `execute_code` + Python `re.sub` 全局替换
  3. 终极方案：terminal用 `head`/`tail` 拆分文件，手动拼接（适用于patch+Python都失败的情况）
  4. 验证：修复后立即用 read_file 确认目标区域内容正确
- 拆分拼接示例：
  ```bash
  head -n 222 文件.md > /tmp/head.txt
  tail -n +224 文件.md > /tmp/tail.txt
  cat /tmp/head.txt 新内容.txt /tmp/tmp.txt > 文件.md
  ```

  - 卷一: Ch1-Ch100（100章）
  - 卷二: Ch101-Ch200（100章）
  - 卷三: Ch201-Ch350（150章）
  - 卷四: Ch351-Ch500（150章）
  - 卷五: Ch501-Ch650（150章）

  if ch_nums:

**发现14：细纲文件章节内容充实度判定（"待补充"不等于空白）**
- 实测数据（《数学太凶猛》700章）：
  - 0-20字（几乎空白）: 476章（68%）- 这些是细纲骨架，正常
  - 21-50字（标题级）: 100章（14%）
  - 51-100字（简短）: 119章（17%）
  - 101-200字（中等）: 4章（0.6%）
  - 200字+（充实）: 1章（Ch1，0.1%）
- 结论：700章中680章不足100字是**细纲阶段的正常状态**（细纲只提供骨架），不代表需要重写
- 判断标准：如果文件名含"细纲"或"7要素"，则低字数是正常现象
- 只有当**正文文件**出现大量低字数时才是质量问题

**发现15：7要素结构细纲的标准化字段集**
- 《数学太凶猛》使用的7要素格式（可用于同类结构检测）：
  - 【核心爽点】【情绪节奏】【爽点位置】【冲突对手】【金手指应用】【章节钩子】【伏笔埋设】
- 字段缺失检测脚本：
  ```python
  required_fields = ['核心爽点', '情绪节奏', '爽点位置', '冲突对手', '金手指应用', '章节钩子', '伏笔埋设']
  for field in required_fields:
      if f'【{field}】' not in block:
          missing.add(field)
  ```
- "（待补充）"占位符正常含义：字段存在但值未填，是细纲阶段填充前的正常状态
- 批量替换公式：
  ```python
  result = re.sub(r'\*\*【冲突对手】\*\*\n- （待补充）',
      '**【冲突对手】**\n- ' + '\n- '.join(conflicts), result)
  # found标志位检查：仅在字段存在时才执行re.sub
  ```

**发现16：伏笔链验证——埋点/回收点/关键词三重核对**
- 伏笔追踪表中的章节号**必须与实际细纲章节内容交叉验证**
- 常见错误模式：
  1. 埋点章节号错位（追踪表写Ch5，实际情节在Ch8）
  2. 回收点章节号断裂（预期Ch250，实际Ch250无相关关键词）
  3. 多条伏笔共享同一回收章节（F004和F007都说Ch300是回收点）
- 验证方法：对每条伏笔，扫描预期埋点±10章和回收点±10章，检查关键词是否真实存在
  ```python
  for fid, (bury, expect_recover, kws) in foreshadowings.items():
      found_recover = None
      for ch in range(expect_recover-10, expect_recover+11):
          block = get_chapter_block(ch)
          if block and any(kw in block for kw in kws):
              found_recover = ch; break
      status = "✅" if found_recover and abs(found_recover-expect_recover)<=15 else "❌"
  ```
- P0级问题：回收断裂（found_recover=None）必须修复

**发现17：爽感四大基石覆盖——不能用"爽感关键词"简单计数**
- 错误方法：直接扫描"打脸/逆袭/天才"等词出现次数
- 正确方法：必须结合**卷定位**判断（逆袭期vs巅峰期，打脸需求完全不同）
- 合理下降判断：
  - 卷3进入IMO国际赛后，公平需求（打脸）从87%→24%是**正常设计**，不是缺陷
  - 读者此时要的是"优越碾压"不是"逆袭打脸"
- 深层羁绊判断（归属需求强度）：
  - 深层羁绊关键词："生死/牺牲/信任/并肩/守护/誓言/决战/保护"
  - 表面温情关键词："婚礼/浪漫/爱/关心/日常/相处"
  - 深层羁绊极低（<5%/卷）是真正的质量问题（卷6大结局情感断崖）

**发现18：爽感模组分布——"常规爽感"占比54%是警告信号**
