# 网文大纲多视角联审流程

对小说大纲/细纲进行三视角联审，输出合订版修改建议与任务清单。

## 核心定位

在每一轮大纲迭代优化之前，用三个不同的"职业视角"对大纲进行诊断：
- **读者视角**：以"能不能吸引我继续读"为唯一标准，测爽感/节奏/伏笔
- **平台编辑视角**：以"这个内容在目标平台能不能火"为标准，测市场定位/节奏偏差/读者留存风险
- **知识库视角**：以"我们知识库里沉淀的方法论是否被遵循"为标准，核对写作规范

三份报告叠加后输出合订版，优先级从高到低排列，附具体任务清单。

## 联审流程

### 第一步：读取知识库核心文件（必须先读）

在启动审核之前，先读以下知识库文件，建立评判基准：

```
concepts/爽感四大基石.md      # 爽感心理机制
concepts/节奏模组.md          # 番茄风vs起点风节奏标准
concepts/读者情绪地图.md      # 2024-2025年读者情绪需求
entities/genres/发疯文学.md   # 2025年热门元素参考
comparisons/传统升级流vs番茄风.md  # 平台风格差异
```

**注**：不需要全部读完整文件，用 `read_file limit=50` 扫前50行建立基准即可。

### 第二步：读取大纲核心文件

```
[项目目录]/D_分卷大纲.md      # 5卷结构总览
[项目目录]/细纲/第一卷细纲.md # 细纲细节
[项目目录]/细纲/第二卷细纲.md
[项目目录]/B_伏笔线.md       # 伏笔追踪表
[项目目录]/C_爽点节奏表.md    # 爽点分布表
[项目目录]/A_人物.md          # 人设一致性参考
[项目目录]/E_时间线.md        # 节奏一致性参考
```

**优先读取顺序**：分卷大纲 → 第一卷细纲 → 第二卷细纲 → 伏笔线 → 爽点节奏表

### 第三步：加载读者审核skill

```bash
skill_view(name="novel-reader-reviewer")
```

使用 `novel-reader-reviewer` skill的四维框架（爽感密度/节奏钩子/人物一致性/伏笔质量）出具**读者审核报告**。

### 第四步：出具番茄编辑报告

不依赖skill，用番茄小说编辑视角自主分析：

**番茄读者核心诉求**（知识库已沉淀）：
1. 快节奏（3章一个小爽点，10章一个大冲突）
2. 甜宠（感情线占比高）
3. 金手指可视化（系统/面板/具体能力要说清楚）
4. 名场面（能在段评里刷屏的台词）

**编辑报告结构**：
- 核心卖点评估（3-5条）
- 节奏问题（番茄风标准 vs 当前差距）
- 人设问题（是否符合目标读者预期）
- 市场定位建议
- 编辑TOP3建议

### 第五步：知识库视角合订

将读者报告 + 编辑报告叠加，去重合并，按**对核心爽感的影响程度**分级：

| 优先级 | 定义 | 对应 |
|--------|------|------|
| **A级** | 影响核心爽感，必须修复 | 读者报告🔴严重问题 + 编辑TOP3 |
| **B级** | 影响节奏和阅读体验，建议修复 | 读者报告🟡建议优化 |
| **C级** | 优化建议，可选择性实施 | 知识库补充项 |

**合订输出格式**：
```
## 合订版修改建议

### A级修改（必须）
| # | 问题 | 涉及文件 | 修改方案 | 验收标准 |

### B级修改（建议）
| # | 问题 | 涉及文件 | 修改方案 |

### C级修改（可选）
| # | 建议 | 涉及文件 |
```

### 第六步：输出任务清单

将所有修改项整理为任务清单，格式：
```
| 序号 | 任务 | 优先级 | 涉及文件 | 预计工作量 |
```

**工作量评估标准**：
- 小：<30min，修改局部段落
- 中：30-60min，涉及多文件调整
- 大：>60min，需重组章节结构

## 实战注意事项（2026-04-19 文道至尊项目验证）

### ⚠️ 子代理并发限制
`delegate_task` 的 `max_concurrent_children` 硬性上限为 **3个**。超过3个任务必须分批：
```python
# ❌ 错误：5个任务会报错
tasks=[goal1, goal2, goal3, goal4, goal5]  

# ✅ 正确：分两批（3+2）
# 第一批：delegate_task(tasks=[goal1, goal2, goal3])
# 第二批：delegate_task(tasks=[goal4, goal5])
```
**批量规划原则**：每批3个任务，并发执行；多余任务单独第二批。

### ⚠️ 章节编号体系冲突陷阱
大型网文项目常存在**两套编号系统**同时运行：
| 系统 | 格式 | 典型特征 |
|------|------|---------|
| 规范文件 | 每卷50章：V1=Ch1-60，V2=Ch61-120 | 参考 CLAUDE.md 或 索引.md |
| 7要素扩写版 | 每卷100+章：V1=Ch1-60，V2=Ch61-120，V3=Ch201-300 | 文件名标注范围与实际内容不匹配 |

**症状**：子代理读取"第三卷细纲（Ch121-Ch180）"但实际内容是 Ch201-300。

**验证方法**：在子代理 context 中强制注明"实际章节范围以文件内容为准，而非文件名标注"。

### ⚠️ patch 增量过少预警
patch 执行后检查 `wc -l` 文件行数增量：
- 增量 < 5行 = 疑似插入位置错误或内容被截断
- 增量异常 → 用 `grep` 验证关键词是否真的出现在正确位置

## 注意事项

- **先读后做**：不要在没读完大纲文件的情况下启动subagent并行修改——会导致返工
- **验证优先**：每一轮修改后用 `grep` / `wc -l` 验证，不轻信"已完成"信号
- **三轮迭代原则**：通常需要3轮才能把大纲打磨到4.5/5分以上
  - Round 1：修复A级问题
  - Round 2：修复B级问题+查漏
  - Round 3：精修名场面/伏笔闭环

## 本skill与 `novel-reader-reviewer` 的区别

| | novel-reader-reviewer | novel-outline-multi-review |
|--|----------------------|---------------------------|
| **范围** | 单一读者视角 | 三视角（读者+编辑+知识库） |
| **输出** | 问题清单 | 问题清单+任务清单+修改方案 |
| **适用** | 单次审核 | 系统性迭代优化 |
| **依赖** | skill自带框架 | 需手动读文件+出具编辑报告 |
